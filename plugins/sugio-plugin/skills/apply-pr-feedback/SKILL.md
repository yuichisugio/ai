# PRフィードバック適用スキル

引数 `$ARGUMENTS` から以下を読み取る:

- **第1引数**: サービス名（`github` または `bitbucket`）
- **第2引数**: PR ID（番号またはURL）

---

## ステップ1: PRレビューデータの取得

サービス名に応じてスクリプトを実行し、出力を取得する。

### GitHub の場合

スキル読み込み時に示された `Base directory for this skill:` のパスを `SKILL_BASE_DIR` として、以下を実行する:

```bash
bash "${SKILL_BASE_DIR}/scripts/fetch_github.sh" $PR_ID
```

**`scripts/fetch_github.sh`** が行うこと:

- `gh` CLI の認証状態を確認（未認証なら `gh auth login` を案内）
- PR基本情報・レビューコメント（承認/却下）・インラインコメント（ファイル・行番号付き）・PR差分を順に出力

### Bitbucket の場合

スキル読み込み時に示された `Base directory for this skill:` のパスを `SKILL_BASE_DIR` として、以下を実行する:

```bash
bash "${SKILL_BASE_DIR}/scripts/fetch_bitbucket.sh" $PR_ID
```

**`scripts/fetch_bitbucket.sh`** が行うこと:

- 以下の環境変数を確認し、未設定の場合はユーザーに案内してから終了:
  - `BITBUCKET_USER` — Bitbucketユーザー名
  - `BITBUCKET_APP_PASSWORD` — アプリパスワード（Settings > App passwords）
  - `BITBUCKET_WORKSPACE` — ワークスペースID（未設定時はgit remoteから自動推定）
  - `BITBUCKET_REPO` — リポジトリスラッグ（未設定時はgit remoteから自動推定）
- PR基本情報・レビューコメント（インライン含む）・PR差分を順に出力

スクリプトがエラーを返した場合は、エラーメッセージをユーザーに提示して対処法を案内する。

---

## ステップ2: レビュー内容の解説

取得したデータをもとに以下の構成で解説する:

### 2-1. 全体サマリー

- レビュアーが何を懸念しているか（1〜3文）
- 承認状態（Approved / Changes requested / Comment only）

### 2-2. 指摘事項の詳細解説（指摘ごとに）

| 項目         | 内容                                           |
| ------------ | ---------------------------------------------- |
| 場所         | ファイルパス・行番号                           |
| 指摘内容     | コメントの要旨                                 |
| 技術的背景   | なぜそれが問題か                               |
| 修正の方向性 | どう直すか（複数ある場合はトレードオフも記載） |

### 2-3. 優先度分類

- **必須対応**: マージブロッカー・バグ・セキュリティ
- **強く推奨**: 設計・保守性の改善
- **任意対応**: スタイル・微細なニュアンス

---

## ステップ3: 複数サブエージェントへの並列依頼

指摘事項の解説が完了したら、**異なるアプローチ**で修正案を生成するサブエージェントを並列で起動する。

### サブエージェントの構成（3体を同時起動）

**Agent A — 保守的アプローチ**（最小変更・既存コードを最大限保持）
**Agent B — 積極的アプローチ**（リファクタリングや設計改善を含む）
**Agent C — バランスアプローチ**（A・Bの中間、テスタビリティ重視）

### 各エージェントへの指示テンプレート

```
以下のPR差分とレビューコメントに基づき、[アプローチ名]で修正案を作成してください。

## PR差分
[gh pr diff / Bitbucket diff の出力を貼付]

## レビューコメント（指摘事項）
[整理済みの指摘リストを貼付]

## あなたの担当アプローチ
[保守的 / 積極的 / バランス]

## 出力形式
各指摘について:
1. 修正後のコードスニペット（diff形式推奨）
2. 修正の根拠（1〜2文）
3. トレードオフ・注意点
4. このアプローチ全体の特徴（最後にまとめ）

コードを実際に変更せず、提案のみを返してください。
```

Task ツールを使い3エージェントを同時起動する（`subagent_type: "generalPurpose"`）。

---

## ステップ4: 修正案の比較と推奨

各エージェントの結果を受け取ったら以下を行う:

### 比較表の作成

| 観点                   | Agent A（保守的） | Agent B（積極的） | Agent C（バランス） |
| ---------------------- | ----------------- | ----------------- | ------------------- |
| 変更量                 | 少                | 多                | 中                  |
| リスク                 | 低                | 高                | 中                  |
| 保守性改善度           | 低                | 高                | 中                  |
| レビュー指摘への対応度 | ...               | ...               | ...                 |

### 推奨案の提示

- **推奨**: Agent [X] のアプローチ
- **理由**: （コードベースの状況・リスク・締め切りなどを加味した根拠）
- **採用時の注意点**: 追加で確認すべきこと

### オプション: ハイブリッド案

A・B・Cの良い部分を組み合わせた最終案があれば提示する。

---

## 注意事項

- **コードの変更はしない**: このスキルは提案フェーズ。実装はユーザーの承認後。
- **環境変数が不足している場合**: 取得できなかった旨を明示し、手動での情報提供を依頼。
- **大量のコメントがある場合**: 優先度「必須対応」のものに絞ってサブエージェントに渡す。
