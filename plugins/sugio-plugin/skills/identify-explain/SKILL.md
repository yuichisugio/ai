# 修正箇所の特定と解説

ユーザーが**具体的な修正内容**（バグ修正・仕様変更・リファクタの方針など）を伝えたあと、**いま手を入れるべき場所を特定して説明する**フェーズです。  
`$ARGUMENTS`について下記を実行してください。

## 前提の確認（足りなければ1問だけ）

- 対象機能・画面・API・再現手順・期待挙動のどれかが曖昧なら、**質問は1つに絞る**。

## やること

1. **依頼の要約**  
   ユーザーが求めている変更を、実装者向けに1〜3文で書く（曲解がなさそうなら短くてよい）。

2. **探索**  
   リポジトリ内を検索・読解し、変更に関わりそうな**ファイル・シンボル（関数・クラス・設定キー）**を洗い出す。推測で決めつけず、根拠（呼び出し元、設定参照、テスト名など）に紐づける。

3. **箇所ごとの解説**（表または箇条書き）  
   各候補について次を書く。
   - **場所**: パス（可能なら概算行・シンボル名）
   - **なぜここか**: 依頼とコードのつながり（データの流れ、分岐、責務）
   - **変更のイメージ**: 何をどう直すと依頼を満たすか（まだコードを書かない）
   - **リスク・副作用**: 他画面／API／互換性への影響があれば短く

4. **優先度**  
   「まず触るべき」「あわせて検討」「今回は外してよさそう」に分類する。

5. **未確定点**  
   リポジトリだけでは断定できないことは「要確認」として列挙する。

## 守ること

- **ユーザーが「実装して」「直して」と明示するまで、ファイルの編集・コミットはしない**（この Skill は調査・解説が役割）。
- 根拠のないファイル束ねを避け、**根拠つき**で候補を並べる。
- 大きな変更が必要そうなら、**小さな段階**に分けられるか一言添える。

## 出力フォーマット（見出しそのまま使用可）

- **依頼の要約**: …
- **修正候補一覧**: （場所／理由／変更イメージ／リスク）
- **触る順（提案）**: …
- **要確認・質問**: …（なければ「なし」）

## **解説方法について**

1. 詳細な処理の流れを解説する
2. 長文で丁寧に解説する
