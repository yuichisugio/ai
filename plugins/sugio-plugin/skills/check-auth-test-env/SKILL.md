# テスト仕様書CSVからE2Eテスト実行スキル

テスト仕様書CSV（結合テストチェックリスト形式）を読み込み、Chrome in Claude でブラウザE2Eテストを実行してMarkdownレポートを出力する。

> **制約**: コードの変更は一切行わない。テスト実行と結果レポートのみを担当する。

---

## CSVフォーマット仕様

このスキルが扱うCSVは以下の構造を持つ:

| 行        | 内容                                                                                       |
| --------- | ------------------------------------------------------------------------------------------ |
| 1行目     | タイトル（例: `結合テストチェックリスト　機能名`）                                         |
| 2行目     | ブランチ名（例: `ブランチ: test/2026-03-23-xxxx`）                                         |
| 3行目     | ヘッダー行（No, 分類, テスト項目, 操作手順, 期待結果, 確認者１, 確認者２, 修正確認, 備考） |
| 4行目以降 | セクション区切り行（`■` で始まる）+ テストケース行                                         |

---

## ステップ1: CSVの解析

`Read` ツールでCSVファイルを読み込み、以下の構造に解析する:

```
{
  title: string,
  branch: string,
  sections: [
    {
      name: string,           // ■ で始まるセクション名
      cases: [
        {
          no: number,
          category: string,   // 分類
          test_item: string,  // テスト項目
          procedure: string,  // 操作手順（改行区切りのステップ）
          expected: string,   // 期待結果（改行区切りの箇条書き）
          notes: string       // 備考
        }
      ]
    }
  ]
}
```

**セクション区切り行の判定**: `No` 列が空でセル内容が `■` で始まる行はセクション名として扱う。

**改行の扱い**: CSVセル内の改行 (`\n`) はステップや箇条書きの区切りとして使用する。

---

## ステップ2: ブラウザセッションの確認

`mcp__claude-in-chrome__tabs_context_mcp` でタブ一覧を取得し、テスト対象アプリが開いているタブを特定する。

- テスト対象URLが既存タブにある場合はそのタブを使用する
- ない場合は `mcp__claude-in-chrome__tabs_create_mcp` で新規タブを作成してアプリにアクセスする
- ユーザーにテスト対象URLを確認してから進む（URLが不明な場合）

---

## ステップ3: テストケースの実行

各テストケースを順番に実行する。**1ケースずつ実行し、結果を記録してから次のケースに進む。**

### 3-1. 操作手順の実行

`操作手順` の各ステップを順番にブラウザで実行する:

| ステップの内容           | 使用ツール                                                                           |
| ------------------------ | ------------------------------------------------------------------------------------ |
| 画面への遷移             | `mcp__claude-in-chrome__navigate`                                                    |
| ボタン・リンクのクリック | `mcp__claude-in-chrome__find` → `ref_id` で操作、またはJSで `.click()`               |
| フォーム入力             | `mcp__claude-in-chrome__form_input`                                                  |
| 画面状態の確認           | `mcp__claude-in-chrome__get_page_text` または `mcp__claude-in-chrome__read_page`     |
| DOM要素の状態確認        | `mcp__claude-in-chrome__javascript_tool`                                             |
| SPA内のルーター遷移      | `mcp__claude-in-chrome__javascript_tool` で `.click()` → `window.location.href` 確認 |

**操作前後の状態取得**:

```
操作前: read_page (filter: "interactive") で操作可能要素を確認
操作後: get_page_text でページ内容を取得して期待結果と照合
```

### 3-2. 期待結果の検証

`期待結果` の各項目を以下の方法で検証する:

| 期待結果の種類       | 検証方法                                                                               |
| -------------------- | -------------------------------------------------------------------------------------- |
| テキストの表示       | `get_page_text` でページテキストを取得し、期待文字列が含まれるか確認                   |
| 要素の表示/非表示    | `javascript_tool` で `querySelector` + `offsetParent` または `display` スタイル確認    |
| ボタンのdisabled状態 | `javascript_tool` で `element.disabled` または `element.hasAttribute('disabled')` 確認 |
| トースト表示         | `get_page_text` でトーストメッセージを確認（トーストは短時間で消えるため素早く確認）   |
| レイアウト確認       | `read_page` でアクセシビリティツリーを取得し、要素の並び順を確認                       |
| CSS スタイル         | `javascript_tool` で `getComputedStyle(element)` を確認                                |

### 3-3. 判定基準

| 判定       | 条件                                               |
| ---------- | -------------------------------------------------- |
| ✅ PASS    | 全ての期待結果が確認できた                         |
| ❌ FAIL    | 1つ以上の期待結果が確認できなかった                |
| ⚠️ SKIP    | 前提条件が整わず実行できなかった（理由を記録）     |
| 🔄 PARTIAL | 一部の期待結果は確認できたが一部は確認できなかった |

---

## ステップ4: テスト実行上の注意点

### トースト通知の確認

トーストは数秒で消えるため、以下の手順で確認する:

```javascript
// 1. 操作を実行する直前にトースト監視を設定
// 2. 操作直後すぐに get_page_text でキャプチャ
// 3. トーストが消えた場合はテスト項目を再実行して再確認
```

もしトーストを見逃した場合は操作を再実行して再確認する。

### ローディング状態の確認

`isLoading` などのローディング状態はページ遷移の直後に確認する必要がある:

```javascript
// ページ遷移直後に disabled 属性を確認
const btn = document.querySelector('[data-testid="login-btn"]');
// または属性名でボタンを特定
const buttons = Array.from(document.querySelectorAll('button'));
const loginBtn = buttons.find((b) => b.textContent.trim() === 'ログイン');
loginBtn?.disabled; // true ならローディング中
```

### 権限が必要なテストケース

権限（管理者/一般ユーザー）が必要なケースは、ブラウザの現在のログインユーザーで対応可能か確認し、対応不可能な場合は SKIP として理由を記録する。

## ステップ5: Markdownレポートの生成

全テストケース実行後、以下のフォーマットでMarkdownレポートを出力する。

```markdown

**テスト仕様書**: {CSVファイル名}
**対象ブランチ**: {branch}
**実行日時**: {YYYY-MM-DD HH:mm}
**テスト環境**: {ブラウザで確認したURL}

---

## サマリー

| 結果       | 件数        |
| ---------- | ----------- |
| ✅ PASS    | {n}         |
| ❌ FAIL    | {n}         |
| ⚠️ SKIP    | {n}         |
| 🔄 PARTIAL | {n}         |

---

| 3   | {テスト項目} | ⚠️ SKIP | {スキップ理由}         |

---
```

---

## 実行フロー

```
CSVを解析 → タブ確認 → [各テストケース] → 操作実行 → 期待結果検証 → 結果記録 → Markdownレポート出力
```

セクション開始時にユーザーへ進捗を報告する（例: `■ チャンネルホーム ログインボタン（7件）の実行を開始します`）。

全件実行後、Markdownレポートをチャットに出力する。レポートはファイル保存不要（チャット出力のみ）。
