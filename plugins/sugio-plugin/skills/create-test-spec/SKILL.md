# テスト仕様書作成スキル

productionブランチと開発ブランチの差分を分析し、変更内容を網羅的にカバーするテスト仕様書をExcelで生成する。

> **制約**: このスキルはテスト仕様書の生成のみを行う。**コードの変更は一切しない。**

---

## ステップ1: 差分の取得（サブエージェント）

Agent ツールで `subagent_type: "general-purpose"` のサブエージェントを起動し、差分を取得する。

スキル読み込み時に示された `Base directory for this skill:` のパスを `SKILL_BASE_DIR` として、サブエージェントに以下のプロンプトで指示する:

```
以下のコマンドをBashツールで実行し、出力をすべてそのまま返却してください。

cd <プロジェクトルート（gitリポジトリのルート）>
bash "${SKILL_BASE_DIR}/scripts/get_diff.sh"

エラーが発生した場合はエラーメッセージをそのまま返却してください。コードの変更や解釈は不要です。
```

**`scripts/get_diff.sh`** が行うこと:

- カレントブランチが production でないことを確認
- `git fetch origin production` でリモートの production を最新化
- `git log --oneline` でコミットログを出力
- `git diff --stat` と `git diff --name-status` で変更ファイル一覧を出力
- `git diff production...<current_branch>` で差分の詳細を出力

スクリプトがエラーを返した場合は、エラーメッセージをユーザーに提示して対処法を案内する。

---

## ステップ2: 差分の分析

取得した差分データを以下の観点で分析する:

### 2-1. 変更の全体像

- 変更されたファイル数、追加/削除行数
- 変更の性質（新機能追加 / バグ修正 / リファクタリング / UI変更 / API変更 など）

### 2-2. 機能単位での変更整理

差分を機能単位でグルーピングし、以下を整理する:

| 項目           | 内容                                             |
| -------------- | ------------------------------------------------ |
| 機能名         | 変更された機能の名前                             |
| 変更種別       | 新規追加 / 修正 / 削除 / リファクタリング        |
| 対象ファイル   | 関連するファイルパス一覧                         |
| 変更の概要     | 何がどう変わったか                               |
| 影響範囲       | この変更が影響する画面・API・データフロー        |
| ロジックの説明 | 変更されたビジネスロジック・条件分岐の詳細な解説 |

### 2-3. テスト観点の洗い出し

各機能の変更について以下の観点でテストケースを網羅的に洗い出す:

- **正常系**: 期待通りの入力で期待通りの結果が得られるか
- **異常系**: 不正入力・境界値・エラーケースの処理
- **境界値**: 最小値/最大値/ゼロ/空文字/null などの境界条件
- **状態遷移**: 初期状態 → 操作 → 遷移後状態
- **表示確認**: UI要素の表示/非表示・スタイル・文言
- **データ永続性**: 保存/リロード後もデータが維持されるか
- **権限・条件分岐**: 権限やフラグによる表示・動作の分岐
- **既存機能への影響（リグレッション）**: 変更が既存機能を壊していないか

## ステップ3: テスト仕様書データの構築

分析結果をもとに、以下のJSON構造でテストデータを構築する。このJSONを中間データとして Write ツールでプロジェクトルートに一時ファイル `_test_spec_data.json` として保存する。

### JSONフォーマット

```json
{
  "title": "<変更概要タイトル>",
  "branch": "<カレントブランチ名>",
  "sections": [
    {
      "name": "<セクション名>　<テスト種別>",
      "cases": [
        {
          "no": 1,
          "category": "<分類タグ>",
          "test_item": "<テスト項目>",
          "procedure": "<操作手順>",
          "expected": "<期待結果>",
          "notes": "<備考>"
        }
      ]
    }
  ]
}
```

### Excelの構成

| 行 | 内容 |
|----|------|
| 1行目 | タイトル行（結合テストの概要とブランチ名を含む） |
| 2行目 | ブランチ名 |
| 3行目 | ヘッダー行（No, 分類, テスト項目, 操作手順, 期待結果, 確認者１, 確認者２, 修正確認, 備考） |
| 4行目以降 | セクション区切り行 + テストケース行 |

### 各カラムの記述ルール

| カラム | 記述ルール |
|--------|-----------|
| No | 通し番号（セクション区切り行はNoなし） |
| 分類 | テストの分類。改行区切りで複数タグを付与可能。例: `"新規作成\n正常系"`, `"API\n異常系\n【バグ修正確認】"` |
| テスト項目 | 1文で何をテストするか。該当ロジックを知らない開発者にも伝わるよう、具体的な機能名・条件・期待動作を含める |
| 操作手順 | 番号付きのステップで記述。画面名・ボタン名・入力値など具体的な操作を記載。`"1. ○○を開く\n2. △△をクリック"` のように改行区切り |
| 期待結果 | 操作後に確認すべき結果を箇条書きで記述。`"・モーダルが閉じる\n・一覧に新規データが表示される"` のように改行区切り。API応答はレスポンス例も含める |
| 確認者１/２ | 空欄（テスト実施者が記入） |
| 修正確認 | 空欄（修正対応時に記入） |
| 備考 | 補足情報。バグ修正確認の場合は `【修正前】○○だった` と修正前の動作を明記 |

### セクション区切り

テストケースは機能単位でセクション区切り行を入れる。JSON の `sections[].name` に `<セクション名>　<テスト種別>` と記載する。

例: `"ユーザー管理　正常系"`, `"ユーザー管理　異常系"`, `"検索機能　表示確認"`

### テストケース記述の重要な注意点

1. **ロジックを知らない開発者が再現できるレベルの具体性**を持たせること
   - 画面パス: `管理画面 > ユーザー一覧` のように階層を明記
   - ボタン名・ラベル: UIに表示される実際の文言を使う
   - 入力値: テストに必要な具体的な値や条件を明記
   - 前提条件: テスト実行前に必要な状態を操作手順の冒頭に記載

2. **バグ修正の変更には `【バグ修正確認】` タグを付与**し、分類に含める。備考に `【修正前】` として修正前の動作を明記する

3. **新規追加の機能には `【新規追加】` タグを付与**し、既存機能との差異がわかるようにする

4. **リグレッション確認**として、変更の影響を受ける可能性がある既存機能のテストケースも含める

5. **JSON内の改行は `\n` で表現する**（Excel生成時にセル内改行に自動変換される）

## ステップ4: Excel生成（サブエージェント）とユーザーへの報告

### 4-1. Excelファイルの生成

Agent ツールで `subagent_type: "general-purpose"` のサブエージェントを起動し、Excelファイルを生成する。

出力ファイル名: `テスト仕様書_<ブランチ名>_<YYYYMMDD>.xlsx`（ブランチ名の `/` は `-` に変換）

サブエージェントに以下のプロンプトで指示する:

```
以下のコマンドをBashツールで実行してください。

cd <プロジェクトルート>
python3 "${SKILL_BASE_DIR}/scripts/create_excel.py" "_test_spec_data.json" "<出力ファイル名>.xlsx"

実行後、出力結果をそのまま返却してください。
成功したら一時ファイル _test_spec_data.json を削除してください。
```

**`scripts/create_excel.py`** が行うこと:

- JSONデータを読み込み、フォーマット済み `.xlsx` ファイルを生成
- ヘッダー行: 紺色背景 (#1F4E79) + 白文字 + 太字
- セクション区切り行: 淡青色背景 (#D6E4F0) + 太字
- データ行: 交互に白/#F2F7FB で色分け、テキスト折り返し有効
- 【バグ修正確認】タグ: 赤文字、【新規追加】タグ: 緑文字
- カラム幅の自動調整、行高のテキスト量に応じた自動調整
- ウィンドウ枠固定（ヘッダー行まで）
- `openpyxl` が未インストールの場合は自動インストール

### 4-2. ユーザーへの報告

以下を報告する:

1. **変更の全体サマリー**（変更ファイル数、変更の性質）
2. **生成したテストケース数**（セクション別の内訳）
3. **テスト観点のカバレッジ**（正常系/異常系/境界値/リグレッションの割合）
4. **特に注意すべきテスト項目**（リスクが高い変更に対するテストケースをハイライト）
5. **Excelファイルのパス**

---

## 注意事項

- **差分が大きすぎる場合**: 変更ファイルが50件を超える場合は、機能単位で主要な変更に絞り、その旨をユーザーに報告する
- **テストケースの粒度**: 1つのテストケースで1つの確認事項を検証する（複数の確認を1ケースに詰め込まない）
- **既存機能への理解が不足する場合**: コードベースを調査して変更の影響範囲を特定し、テストケースに反映する。不明な点はユーザーに確認する
