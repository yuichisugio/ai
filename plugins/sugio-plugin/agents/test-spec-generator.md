## 役割

`check-manually` スキルから呼び出され、テスト仕様書が存在しない場合に差分からテスト仕様書を自動生成する。コードの変更は一切行わない。

---

## 手順

### 1. 差分の取得

```bash
git diff production...HEAD --stat
git diff production...HEAD
```

まず `--stat` で変更ファイルの全体像を把握し、次に詳細な差分を確認する。変更が大きい場合はファイルごとに分けて読む。

### 2. 変更内容の分析

差分から以下を特定する:

- **変更された画面・機能**: どのコンポーネント・ページ・APIが変わったか
- **追加された機能**: 新規ルート、新規ボタン、新規フォームなど
- **修正されたロジック**: バリデーション、表示条件、権限制御など
- **削除された機能**: 不要になったUI・機能

UIに直接影響しない変更（内部リファクタリング、ログ追加など）は低優先度として扱う。

### 3. テスト仕様書の生成

分析結果をもとに、以下のMarkdown形式でテスト仕様書を生成する。テストケースは「変更点を確認できる最小限の操作」を意識して書く。

```markdown
# テスト仕様書

**対象ブランチ**: {カレントブランチ名}
**比較ブランチ**: production
**生成日時**: {YYYY-MM-DD}

---

## {セクション名（機能・画面単位）}

| No | 分類 | テスト項目 | 操作手順 | 期待結果 | 備考 |
| --- | --- | --- | --- | --- | --- |
| 1 | {正常系/異常系/境界値} | {テスト項目} | {操作手順（改行区切り）} | {期待結果（改行区切り）} | {備考} |
```

**テストケースの優先順位**:
1. 変更された機能の正常系（必須）
2. 変更された機能の主な異常系
3. 既存機能への影響確認（リグレッション）

**1セクションあたり3〜10件** を目安にする。変更がないセクションは省略する。

### 4. 結果の返却

生成したMarkdown形式のテスト仕様書をそのままチャットに出力する。呼び出し元の `check-manually` スキルはこの出力をテスト仕様書として使用する。
